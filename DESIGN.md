# Lyru OMS Design

## 목적

Lyru OMS는 1인 가게 사장님이 매일 주문과 요청사항을 놓치지 않게 돕는 운영 도구다. 디자인은 화려한 마케팅 페이지가 아니라, 바쁜 제작 과정 중에도 빠르게 확인하고 판단할 수 있는 조용한 작업 화면이어야 한다.

핵심 사용자는 프리미엄 수제 곶감 디저트와 화과자를 만드는 1인 운영자다. 화면은 사장님의 기억 부담을 줄이고, 오늘 처리해야 할 주문, 희망 발송일, 변경 요청, 미확인 체크를 즉시 드러내야 한다.

## 디자인 원칙

1. **오늘 할 일을 먼저 보여준다.** 첫 화면은 통계보다 오늘 제작/발송/확인해야 할 주문을 우선한다.
2. **누락 위험을 조용하지만 분명하게 표시한다.** 미확인 변경, 필수 정보 누락, 생산량 초과는 색과 문구로 즉시 알아볼 수 있어야 한다.
3. **모바일 작업을 기본값으로 본다.** 제작 중 휴대폰으로 확인하고 체크할 수 있도록 터치 영역과 정보 밀도를 설계한다.
4. **자동화보다 검증 가능성을 우선한다.** 초기 제품은 수동 입력과 가져오기를 허용하되, 입력값의 출처와 변경 이력을 명확히 남긴다.
5. **프리미엄 브랜드 감성을 해치지 않는다.** 운영 도구라도 화이트, 골드, 브라운, 다크 네이비 기반의 차분한 톤을 유지한다.

## 정보 구조

### 핵심 화면

- **오늘 작업**: 오늘 발송, 제작 준비, 확인 필요, 미확인 변경 요청을 한곳에 모은다.
- **주문 대장**: 모든 주문을 검색, 필터, 정렬하는 기본 목록이다.
- **주문 상세**: 고객 정보, 상품, 수량, 희망 발송일, 요청사항, 변경 이력, 상태 변경을 다룬다.
- **일정/물량**: 날짜별 주문 수량과 생산 가능량 초과 여부를 보여준다.
- **가져오기/내보내기**: CSV 또는 엑셀 기반 주문 반영과 백업을 처리한다.

### 첫 화면 우선순위

1. 확인 필요 주문
2. 오늘 발송 주문
3. 오늘 제작 준비 주문
4. 미확인 변경 요청
5. 생산량 초과 위험 날짜

| Token | Color | Use |

### 사용 규칙

- 골드는 주요 버튼과 선택 상태에만 제한적으로 사용한다.
- 위험 상태는 빨간색만으로 의존하지 말고 텍스트와 아이콘을 함께 사용한다.
- 배경은 순백보다 따뜻한 화이트를 사용해 프리미엄 디저트 브랜드 감성을 유지한다.
- 다크 네이비는 넓은 배경보다 상단 바, 강조 패널, 중요한 요약 영역에 제한적으로 쓴다.

## 타이포그래피

### 폰트

- **기본 UI**: Pretendard, Inter, system-ui, sans-serif
- **숫자/코드성 데이터**: JetBrains Mono 또는 ui-monospace

### 계층

| Style | Size | Weight | Use |
|-------|------|--------|-----|
| Display | 28-32px | 700 | 오늘 작업 화면의 주요 제목 |
| Title | 20-24px | 700 | 화면 제목, 주요 패널 제목 |
| Section | 16-18px | 700 | 목록 그룹, 카드 제목 |
| Body | 14-16px | 400 | 기본 텍스트 |
| Label | 12-13px | 600 | 필터, 상태, 입력 라벨 |
| Dense | 12-13px | 400 | 표 보조 정보, 이력 |

### 문구 원칙

- 사장님이 바로 이해할 수 있는 한국어를 쓴다.
- "Pending", "Processing" 같은 내부 용어 대신 "확인 필요", "제작 준비", "발송 준비"를 사용한다.
- 경고 문구는 원인과 행동을 같이 말한다. 예: "미확인 변경 요청이 있어 발송 완료 전 확인이 필요합니다."

## 레이아웃

### 데스크톱

- 좌측에는 주요 메뉴를 두고, 우측은 작업 영역으로 사용한다.
- 주문 목록은 표 형태를 기본으로 하되, 상태와 발송일이 한눈에 보이게 한다.
- 상세 정보는 오른쪽 패널 또는 별도 상세 화면으로 열어 목록 맥락을 잃지 않게 한다.

### 모바일

- 하단 탭 또는 상단 간결 메뉴로 오늘 작업, 주문, 일정에 접근한다.
- 주문 카드는 고객명, 발송일, 상태, 요청 경고를 첫 줄에서 파악하게 한다.
- 상세 화면의 주요 액션은 하단 고정 영역에 둔다.
- 터치 대상은 최소 44px 높이를 확보한다.

## 컴포넌트

### 주문 카드

주문 카드 또는 목록 행은 다음 정보를 안정적으로 보여준다.

- 고객명
- 주문 출처
- 희망 발송일
- 상품 요약과 수량
- 현재 상태
- 미확인 요청 여부
- 생산/발송 위험 여부

### 상태 배지

| State | Label | Tone |
|-------|-------|------|
| New | 신규 | info |
| Needs Check | 확인 필요 | warning |
| Prep | 제작 준비 | brown |
| Ready | 발송 준비 | gold |
| Shipped | 발송 완료 | success |
| Canceled | 취소 | muted |

### 요청 체크리스트

- 체크 항목은 주문 상세에서 가장 빠르게 접근할 수 있어야 한다.
- 미완료 항목은 발송 완료 액션 전에 다시 드러난다.
- 항목은 짧은 문장으로 기록하고, 필요하면 요청 채널을 같이 표시한다.

### 날짜/물량 표시

- 날짜별 주문 수량, 제작 수량, 생산 가능량을 함께 보여준다.
- 생산 가능량 초과는 danger, 근접은 warning으로 표시한다.
- 달력보다 "오늘/내일/이번 주" 작업 리스트가 우선이다.

### 가져오기

- CSV/엑셀 가져오기는 단계형 흐름으로 구성한다.
- 가져온 주문 중 필수 정보가 누락된 항목은 저장 전 수정하게 한다.
- 중복 가능성이 있는 주문은 확인 필요 상태로 표시한다.

## 접근성

- 텍스트와 배경 대비는 WCAG AA 수준을 목표로 한다.
- 상태 정보는 색상만으로 전달하지 않는다.
- 키보드 탐색과 스크린 리더 라벨을 기본 제공한다.
- 모바일에서 버튼과 체크박스는 손가락 조작에 충분한 크기를 가진다.

## 반응형 기준

| Range | Behavior |
|-------|----------|
| `< 768px` | 모바일 카드 목록, 하단 액션, 단일 컬럼 상세 |
| `768-1199px` | 2컬럼 또는 목록+상세 패널 |
| `>= 1200px` | 좌측 내비게이션, 중앙 목록, 우측 상세/요약 패널 |

## 금지 사항

- 첫 화면을 마케팅 랜딩 페이지처럼 만들지 않는다.
- 그래프와 통계를 주문 확인보다 먼저 보여주지 않는다.
- 화려한 장식, 과도한 그라디언트, 복잡한 애니메이션을 사용하지 않는다.
- 자동 연동이 없는 상태에서 자동 수집된 것처럼 보이는 문구를 쓰지 않는다.
- 생산량 초과나 미확인 요청 같은 위험 신호를 작은 보조 텍스트로 숨기지 않는다.

## v1 디자인 완료 기준

- 사장님은 첫 화면에서 오늘 확인해야 할 주문과 위험 요소를 알 수 있다.
- 주문 목록에서 발송일, 상태, 요청 경고를 빠르게 스캔할 수 있다.
- 주문 상세에서 요청사항 체크와 상태 변경이 1-2번의 조작으로 가능하다.
- 모바일 화면에서 주문 확인과 체크가 데스크톱 기능의 축소판이 아니라 실제 작업 흐름으로 동작한다.
- 브랜드 톤은 프리미엄 수제 디저트에 어울리지만, 운영 도구로서 정보 전달을 방해하지 않는다.
