## Phase Details

### Phase 1: 주문 표준화 MVP

**Goal:** 사장님이 여러 채널의 주문/문의 원문을 붙여넣고, 가능한 항목을 구조화해 정보 부족과 확인 필요 주문을 빠르게 볼 수 있다.
**Mode:** mvp
**UI hint:** yes

**Requirements:** ORD-03, ORD-04, ORD-05, ORD-06, REQ-05, REQ-06, REQ-07, REQ-08, REQ-09, REQ-10, REQ-11, REQ-12, REQ-13, DATA-03, DATA-04, DATA-05

**Success Criteria:**

1. 사용자는 카카오톡, 인스타그램, 네이버 톡톡 등에서 받은 주문/문의 원문을 붙여넣고 저장할 수 있다.
2. 시스템은 알려진 주문서 라벨과 필드별 연관 키워드를 기준으로 고객명, 연락처, 주문 내용, 수량, 희망일 등 가능한 항목을 채운다.
3. 사용자는 추출된 항목을 수동으로 수정할 수 있고 원문은 저장된 기준 기록으로 남는다.
4. 필수 정보가 부족하거나 대량/맞춤 요청으로 보이는 주문은 정보 부족 또는 확인 필요로 사유를 분리해 표시하되, 둘은 같은 주의 경고 레벨로 묶어 보여주고 상태는 자동으로 확인필요가 된다.
5. 수량이 대량 주문 기준 이상이면 확인 필요로 표시된다.
6. 사용자는 관리 모달에서 정보 부족 판정에 사용할 필수 항목과 대량 주문 기준 수량을 조정할 수 있다.
7. 설정이 바뀌어도 기존 정리 완료 주문은 자동으로 확인필요로 되돌아가지 않는다.
8. 사용자는 예외적으로 주문/문의를 직접 확인필요 상태로 바꿀 수 있다.
9. 같거나 비슷한 원문이 이미 있으면 중복 가능성을 경고하되 저장은 허용한다.
10. 원문 수정이나 재파싱이 있어도 사장님이 수동 수정한 값은 자동으로 덮어쓰지 않고, 차이가 있으면 무채색 점과 터치/호버 설명으로만 표시한다.
11. Phase 1 검증 기간에는 개발 방향 결정을 위해 사장님에게 확인해야 할 질문을 주문 수집 화면 상단의 접이식 임시 쪽지 영역에서 볼 수 있고, 답변은 앱에 저장하지 않는다.
12. 필수 정보가 빠지거나 파싱이 실패해도 원문 저장은 차단되지 않는다.

### Phase 2: 주문 대장과 요청 변경 관리

**Goal:** 정리된 주문을 대장으로 관리하고 흩어진 고객 요청을 주문에 붙여 미확인 변경을 놓치지 않는다.
**Mode:** mvp
**UI hint:** yes

**Requirements:** ORD-01, ORD-02, REQ-01, REQ-02, REQ-03, STS-01, STS-02

**Success Criteria:**

1. 사용자는 정리 완료된 주문을 주문 대장에 등록하고 상태별로 볼 수 있다.
2. 사용자는 주문별 요청사항 체크리스트를 만들고 완료 표시할 수 있다.
3. 사용자는 변경 요청의 채널, 변경 전후 내용, 확인 여부를 기록할 수 있다.
4. 사용자는 CSV 또는 엑셀 주문을 가져와 주문 대장에 반영할 수 있다.

### Phase 3: 발송일과 생산량 보기

**Goal:** 희망 발송일 기준으로 오늘/이번 주 물량과 위험 날짜를 본다.
**Mode:** mvp
**UI hint:** yes

**Requirements:** SCH-01, SCH-02, SCH-03, SCH-04

**Success Criteria:**

1. 사용자는 오늘, 내일, 이번 주 발송 주문을 빠르게 확인할 수 있다.
2. 사용자는 날짜별 주문 수량과 제작 물량을 볼 수 있다.
3. 시스템은 하루 생산 가능량을 넘는 날짜를 위험 상태로 표시한다.
4. 사용자는 발송일 기준으로 주문을 정렬하고 필터링할 수 있다.

### Phase 4: 출고 전 검수와 이력

**Goal:** 발송 완료 전에 미확인 요청과 변경 이력을 확인한다.
**Mode:** mvp
**UI hint:** yes

**Requirements:** REQ-04, STS-03, STS-04, DATA-01, DATA-02

**Success Criteria:**

1. 사용자는 오늘 처리해야 할 주문만 모아 보는 작업 화면을 사용할 수 있다.
2. 발송 완료 처리 전에 미확인 요청사항이 있으면 경고가 표시된다.
3. 사용자는 주문 상세에서 상태와 주요 변경 이력을 볼 수 있다.
4. 사용자는 주문 데이터를 파일로 내보내 백업할 수 있다.

### Phase 5: 모바일 운영 최적화

**Goal:** 휴대폰에서 주문 확인, 검색, 상태 변경을 빠르게 수행한다.
**Mode:** mvp
**UI hint:** yes

**Requirements:** MOB-01, MOB-02, MOB-03

**Success Criteria:**

1. 모바일 화면에서 주문 목록, 상세, 요청사항 체크가 끊기지 않고 동작한다.
2. 주요 상태 변경 버튼은 한 손 조작에 맞는 크기와 위치를 가진다.
3. 사용자는 고객명, 연락처, 주문 출처, 발송일로 주문을 찾을 수 있다.
4. 모바일 목록과 상세 화면의 정보 밀도는 작업 중 빠른 확인에 적합하다.

## Definition of Done

- 모든 v1 요구사항은 하나의 Phase에 매핑되어야 한다.
- 각 Phase는 동작 가능한 사용자 흐름을 남겨야 한다.
- 한국어 UI 문구와 문서는 실제 1인 가게 사장님이 이해할 수 있는 표현을 사용해야 한다.
- 자동화보다 누락 방지와 수동 검증 가능성을 우선해야 한다.
