# 주문 목록 운영 뼈대 설계

## 배경

Lyru OMS Phase 1은 사장님이 주문/문의 원문을 붙여넣고, 가능한 주문 항목을 파싱하고, 추출값을 보정한 뒤 저장하는 흐름을 제공한다. 다음 핵심 뼈대는 입력 경로를 늘리는 것이 아니라, 이미 저장된 주문을 사장님이 매일 믿고 훑고 고칠 수 있는 운영 대장으로 만드는 것이다.

이 설계는 Phase 2 방향의 첫 번째 작은 세로 조각이다. 저장된 주문을 주문 목록에서 운영하고, 최소 상태값을 붙이며, 단순한 변경 요청 확인 흐름을 추가한다. 수동 주문 추가, CSV/엑셀 가져오기, 여러 건 변경 이력은 이번 범위에서 제외한다.

## 목표

- 주문 입력은 기존 원문 붙여넣기 흐름으로 단일화한다.
- 주문 목록을 기본 운영 대장으로 사용할 수 있게 한다.
- 기본 정렬은 희망일 기준으로 두고, 최근 등록순과 수량 많은 순도 제공한다.
- 상태값은 일상 운영에 필요한 최소 4개로 제한한다.
- 변경 요청을 간단 메모와 확인 체크로 관리해 미확인 변경을 목록에서 놓치지 않게 한다.
- 기존 Phase 1 저장 데이터는 hydrate 단계에서 안전하게 보정한다.

## 제외 범위

- 새 수동 주문 등록 화면은 만들지 않는다.
- CSV/엑셀 가져오기는 포함하지 않는다.
- 여러 건 변경 요청 이력은 만들지 않는다.
- 상태 변경 이력이나 감사 로그는 포함하지 않는다.
- 출고 전 검수 체크리스트는 포함하지 않는다.

## 사용자 흐름

1. 사장님이 기존 주문 수집 화면에 주문/문의 원문을 붙여넣는다.
2. 시스템이 원문을 파싱하고 확인 필요 여부를 평가한 뒤 주문을 저장한다.
3. 저장된 주문은 주문 목록에 기본적으로 희망일 빠른 순으로 표시된다.
4. 사장님은 목록에서 상태, 희망일, 고객/상품 요약, 주요 배지를 훑는다.
5. 사장님은 주문을 열어 상태, 주문 필드, 고객 요청사항, 내부 메모, 변경 요청 메모를 수정한다.
6. 변경 요청이 있는데 아직 확인되지 않았다면 목록과 상세에 별도 경고가 표시된다.

## 데이터 모델

`CapturedOrder`는 기존 Phase 1 필드를 유지하고 다음 두 필드를 추가한다.

```ts
changeRequestNote: string;
changeRequestConfirmed: boolean;
```

`changeRequestNote`는 주문 저장 이후 고객이 요청한 변경 내용을 기록한다. `changeRequestConfirmed`는 사장님이 그 변경 요청을 확인했는지 기록한다. 기존 `customerRequestNote`는 주문 당시 고객 요청사항, `ownerMemo`는 사장님 내부 메모, 새 `changeRequestNote`는 저장 이후 발생한 변경 요청으로 역할을 분리한다.

상태값은 Phase 1 수집 상태에서 운영 상태로 전환한다.

```ts
['신규', '확인 필요', '제작 준비', '발송 완료']
```

기존 저장 데이터의 상태는 hydrate 단계에서 다음처럼 매핑한다.

| 기존 상태 | 새 상태 |
| --- | --- |
| `수집` | `신규` |
| `확인필요` | `확인 필요` |
| `정리 완료` | `제작 준비` |

새 변경 요청 필드가 없는 기존 주문은 다음 기본값으로 hydrate한다.

```ts
changeRequestNote: '';
changeRequestConfirmed: false;
```

## 정렬

주문 목록의 기본 정렬은 `희망일 빠른 순`이다. 목록 헤더에는 다음 정렬 옵션을 추가한다.

- `희망일 빠른 순`
- `최근 등록순`
- `수량 많은 순`

정렬 책임은 `OrderList`가 아니라 `App`에 둔다. `App`이 `sortMode` 상태를 갖고 `useMemo`로 `displayOrders`를 계산해 `OrderList`에 전달한다. 이렇게 하면 저장 순서, 선택된 주문 조회, 화면 표시 순서가 섞이지 않는다.

정렬 규칙은 다음과 같다.

- `희망일 빠른 순`: 비교 가능한 희망일이 있는 주문을 먼저 표시하고, 날짜가 빠른 주문을 위에 둔다. 희망일을 비교할 수 없는 주문은 아래로 보내고 최근 등록순으로 보조 정렬한다.
- `최근 등록순`: `createdAt`이 최신인 주문을 먼저 표시한다.
- `수량 많은 순`: `quantityCandidates`가 있으면 그 값을 우선 사용하고, 없으면 `quantity` 문자열에서 비교 가능한 숫자를 사용한다. 비교 가능한 수량이 없는 주문은 아래로 보내고 최근 등록순으로 보조 정렬한다.

## 목록 UI

`OrderList`는 주문을 빠르게 훑는 화면이다. 기존 카드형/목록형 보기 토글은 유지하고, 별도의 정렬 선택 컨트롤을 추가한다. 보기 방식은 정보 밀도를 조절하고, 정렬 방식은 표시 순서를 조절한다.

각 주문 행은 다음 정보를 드러낸다.

- 출처
- 운영 상태
- D-day 또는 희망일 상태
- 고객명 또는 fallback
- 상품/수량 요약
- 희망일과 수령 방식
- 주요 배지

배지는 혼합형으로 사용한다.

- `확인 필요`: 기존 Phase 1 평가 결과가 주의 상태이거나, 확인 사유 또는 부족 필드가 있을 때 표시한다.
- `변경 확인 필요`: `changeRequestNote`가 비어 있지 않고 `changeRequestConfirmed`가 `false`일 때 표시한다.
- `고객 요청 있음`: `customerRequestNote`가 비어 있지 않을 때 표시한다.
- `내부 메모 있음`: `ownerMemo`가 비어 있지 않을 때 표시하되, 목록에는 메모 내용을 노출하지 않는다.

`변경 확인 필요`는 `확인 필요`와 분리한다. 고객 변경 요청은 파싱 실패나 정보 부족이 아니라 운영자가 직접 확인해야 하는 별도 작업이기 때문이다.

## 상세 UI

`OrderDetail`은 기존 모달 패턴을 유지하되, 역할을 "추출값 확인"에서 "주문 편집 상세"로 넓힌다.

권장 섹션 순서는 다음과 같다.

1. 확인 필요 요약: 기존 정보 부족과 확인 필요 사유를 먼저 보여준다. 미확인 변경 요청이 있으면 `변경 요청 확인 필요`를 별도 항목으로 표시한다.
2. 변경 요청: `changeRequestNote` textarea와 `changeRequestConfirmed` checkbox를 둔다. 변경 요청 메모가 비어 있으면 체크박스는 숨기거나 비활성화한다.
3. 주문 정보 편집: 기존 편집 필드를 유지하고, 고객 요청사항과 내부 메모를 계속 분리한다.
4. 주문/문의 원문: 원문 편집은 유지하되, 기준 기록으로 다루기 위해 화면 하단 또는 덜 두드러지는 영역으로 배치한다.

원문 수정 시 기존 재파싱 동작을 유지한다. 파서가 새 추출값을 제안할 수 있지만, 사장님이 직접 수정한 필드는 자동으로 덮어쓰지 않는다.

## 상태 동작

사장님은 다음 4개 상태 중 하나를 직접 선택할 수 있다.

- `신규`
- `확인 필요`
- `제작 준비`
- `발송 완료`

`evaluateOrder`는 정보 부족이나 확인 필요 사유가 있을 때 주문을 `확인 필요`로 전환할 수 있다. 다만 기존 Phase 1 원칙처럼 설정 변경만으로 이미 `제작 준비`나 `발송 완료`로 둔 주문을 다시 `확인 필요`로 되돌리지는 않는다.

미확인 변경 요청은 상태를 강제로 `확인 필요`로 바꾸지 않는다. 대신 별도 배지와 상세 요약으로 보여준다. 이렇게 해야 운영 상태와 변경 요청 확인 작업의 의미가 섞이지 않는다.

## 테스트

도메인 테스트:

- 운영 상태값이 4개인지 확인한다.
- 기존 저장 상태가 새 운영 상태로 hydrate되는지 확인한다.
- 변경 요청 필드가 없는 기존 주문이 안전한 기본값으로 hydrate되는지 확인한다.
- 수량 정렬 비교값이 `quantityCandidates`, `quantity` 문자열, 수량 미정 케이스에서 안정적으로 계산되는지 확인한다.

컴포넌트 테스트:

- 주문 목록 기본 정렬이 희망일 빠른 순인지 확인한다.
- 정렬 방식을 최근 등록순이나 수량 많은 순으로 바꾸면 표시 순서가 바뀌는지 확인한다.
- 미확인 변경 요청이 있는 주문에 `변경 확인 필요` 배지가 보이는지 확인한다.
- 변경 요청을 확인 처리하면 해당 배지가 사라지는지 확인한다.
- 상세의 상태 선택지가 4개 운영 상태로 제한되는지 확인한다.

수동 검증:

- 데스크톱에서 정렬 선택, 보기 토글, 목록 선택, 상세 편집이 함께 동작하는지 확인한다.
- 모바일 폭에서 정렬 선택과 보기 토글이 겹치거나 과밀해지지 않는지 확인한다.
- 기존 localStorage 데이터가 새 상태와 변경 요청 기본값으로 깨지지 않고 표시되는지 확인한다.

## 열린 결정

없음. 수동 주문 추가, CSV/엑셀 가져오기, 변경 요청 이력, 상태 이력은 의도적으로 다음 범위로 미룬다.
