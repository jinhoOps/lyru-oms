# Phase 1 Rough Order Intake MVP Design

## 목적

Phase 1 rough order intake MVP는 사장님이 카카오톡 채널, 네이버 톡톡, 인스타그램 DM, 스마트스토어 문의 등에서 받은 주문/문의 원문을 잃지 않고 저장한 뒤, 가능한 필드와 확인해야 할 내용을 빠르게 정리하도록 돕는다.

이번 설계는 AI API를 런타임에 호출하지 않는다. 대신 개발 단계에서 정리한 메뉴 사전, 용도 키워드, 수량/날짜/택배 규칙을 앱 내부 규칙으로 실행한다. 입력 UX는 초안 수준을 허용하되, 화면은 복잡한 분석 도구처럼 보이지 않고 사장님이 다음에 확인할 일을 자연스럽게 알 수 있어야 한다.

## 설계 방향

추천 접근은 메뉴 사전 기반 규칙 파서다.

- 라벨 기반 주문서 입력은 기존 매크로 라벨을 우선 신뢰한다.
- 상담/의사표시형 자연어 입력은 상품명, 용도, 수량 후보, 날짜, 수령 방식 신호를 찾는다.
- 애매한 값은 자동 확정하지 않고 후보로 저장하며 확인할 내용으로 표시한다.
- 원문은 항상 기준 기록으로 보존한다.
- AI API 파싱, 자동 채널 연동, 메뉴 관리 화면은 Phase 1 범위 밖이다.

## 입력 유형

### 주문서형

사장님 매크로를 따라 작성된 입력이다. 다음 라벨을 인식한다.

- 성함, 이름, 고객명
- 연락처, 전화번호, 휴대폰
- 주문 내용, 주문상품, 상품
- 수량, 개수, 주문 수량
- 선물 용도, 용도
- 픽업/택배, 수령 방식, 수령방법
- 픽업 날짜 및 시간, 희망일, 희망 날짜, 배송 날짜
- 픽업 시간, 방문 시간
- 택배 주소, 주소, 배송지
- 견과류 알레르기 유무, 알레르기
- 추가 옵션, 보자기/노리개/꽃
- 요청사항, 기타 요청사항, 메모

주문서형에서는 라벨 오른쪽 값을 해당 필드에 채운다. 비어 있는 라벨은 저장을 막지 않고 정보 부족 판정에만 사용한다.

### 상담/의사표시형

주문서 작성 전 고객이 보낸 자연어 또는 짧은 메모다.

예:

```text
결혼식 답례품
대추야자 2구/9구
180개 / 20개 구매 의사 먼저 밝히려고 합니다!!!
```

이 입력은 다음처럼 처리한다.

- `결혼식 답례품`은 선물 용도로 채운다.
- `대추야자 2구/9구`는 주문 내용으로 채운다.
- 상품 후보는 내부 후보 목록에 저장한다.
- `180개 / 20개`는 표시용 수량과 내부 수량 후보로 분리한다.
- 후보가 여러 개이므로 상태는 확인필요가 된다.

상담/의사표시형은 고객명이나 연락처가 없어도 저장한다. Phase 1의 우선순위는 완성 주문만 받는 것이 아니라 흩어진 원문을 놓치지 않는 것이다.

## 메뉴 사전

Phase 1에서는 메뉴 정보를 코드 내장 사전으로 둔다. 사장님이 메뉴를 직접 관리하는 화면은 후속 브레인스토밍으로 분리한다.

사전은 다음 정보를 가진다.

- 대표 메뉴명
- alias 또는 검색 키워드
- 구성 수량
- 최소 주문 조건
- 택배 가능 여부 또는 확인 필요 여부
- 시즌 판매/품절 메모
- 옵션 여부

초기 상품군은 다음 메뉴를 포함한다.

- 상견례세트 A/B/C
- 화과자 2구, 4구, 6구, 8구, 9구
- 곶감밀푀유 8구, 9구, 선물세트
- 시그니처 곶감말이
- 곶감말이 정과세트, 화과자세트, 단지세트
- 대추야자 오동나무 9구 세트
- 대추야자 10구 세트
- 대추야자 프리미엄 15구 세트
- 오란다&모나카 올인박스
- 모나카견과칩 라운드박스
- 퓨전찹쌀떡
- 밤화과자
- 생화쌀다쿠아즈케이크
- 보자기 포장
- 노리개 데코 포장

용도 키워드는 `상견례`, `답례품`, `결혼식`, `기업답례품`, `부모님선물`, `스승님`, `집들이`, `어버이날`, `연말선물`, `크리스마스선물`, `첫만남`, `결혼기념`, `인사`, `단체`, `견적`, `구매 의사`를 기본값으로 둔다.

## 수량과 상품 후보

표시용 값과 내부 후보를 분리한다.

- `주문 내용`: 사장님이 보는 대표 문자열
- `상품 후보`: 내부 상품 매칭 후보 목록
- `수량`: 사장님이 보는 대표 문자열
- `수량 후보`: 내부 수량 후보 목록

예:

```text
주문 내용: 대추야자 2구/9구
상품 후보: 대추야자 오동나무 9구 세트, 대추야자 10구 세트, 대추야자 프리미엄 15구 세트
수량: 180개 / 20개 후보
수량 후보: 180, 20
```

`9구`, `15구`, `6개입`, `8구/9구` 같은 숫자는 상품 구성 숫자로 본다. 주문 수량 후보에서는 제외한다. 전화번호, 가격, 배송비도 수량 후보에서 제외한다.

수량 후보가 1개면 표시용 수량에 확정값처럼 채운다. 수량 후보가 2개 이상이면 표시용 수량에는 원문에 가까운 후보 문자열을 넣고 확인할 내용에 `수량 후보 여러 개`를 추가한다.

## 주문 수량 조건

기존 대량 주문 기준은 `주문 수량 조건`으로 확장한다. 최소 주문 조건과 대량 기준을 모두 다룬다.

### 최소 주문 조건

메뉴 사전에 상품별 최소 주문 조건을 둔다.

- 화과자 2구: 5세트 이상
- 화과자 4구: 2세트 이상

상품이 명확히 매칭되고 주문 수량이 최소 조건보다 적으면 확인할 내용에 `최소 주문 조건 확인`을 추가한다. 상품 매칭이 애매하면 저장을 차단하지 않고 `상품 후보 여러 개` 또는 `최소 주문 조건 확인`으로 남긴다.

### 대량 기준

대량 기준 기본값은 실수량 40구 이상이다.

실수량은 `구성 수량 x 주문 세트 수`로 계산한다.

- 화과자 9구 5세트: 45구, 대량
- 대추야자 15구 3세트: 45구, 대량
- 화과자 4구 5세트: 20구, 대량 아님

구성 수량이나 세트 수가 애매하면 대량 여부를 확정하지 않고 확인할 내용으로 남긴다.

행사성 또는 견적성 키워드가 있으면 실수량과 무관하게 확인필요로 둔다.

## 날짜와 D-Day 표시

명시 날짜만 자동 확정한다.

자동 추출 대상:

- `7/5`
- `7월 5일`
- `2026-07-05`
- `7월 5일 오후 2시`
- `토요일 11시`, 단 날짜 기준이 명확한 경우에만 사용

상대 날짜는 자동 확정하지 않는다.

- 이번 주 금요일
- 다음 주말
- 어버이날 전
- 결혼식 전날
- 최대한 빨리

상대 날짜는 `날짜 확인 필요`로 표시하고, 원문 표현을 확인할 내용에 남긴다.

D-Day 표시는 다음 규칙을 따른다.

- 미래 날짜: `D-3`
- 오늘이고 시간이 있음: `오늘 14:00`
- 오늘이고 시간이 없음: `오늘`
- 지난 날짜: `D+1`
- 날짜 확정 불가: `날짜 확인 필요`

D-Day 배지는 터치, hover, focus 시 원래 날짜를 보여준다.

예:

```text
2026년 7월 5일
2026년 7월 5일 14:00
원문 표현: 이번 주 금요일
```

## 수령 방식과 택배 규칙

`택배`, `배송`, `주소`, `배송지`가 있으면 수령 방식 후보를 택배로 둔다. `픽업`, `방문`이 있으면 픽업 후보로 둔다. 둘 다 있거나 부정 표현이 섞이면 확정하지 않는다.

택배 신호가 있으면 사장님이 확인해야 하므로 확인할 내용에 `택배 가능 여부`를 추가한다. 톡톡이나 카카오톡으로 들어온 내용이어도 택배는 사장님 확인 대상으로 둔다.

정보 부족 기본 규칙은 다음과 같다.

- 택배인데 주소가 없으면 `택배 주소`
- 픽업 또는 택배인데 희망일이 없으면 `희망일`
- 픽업 시간이 없어도 정보 부족으로 보지 않는다.

## 필수 항목 기본값

관리 모달에서 필수 여부를 바꿀 수 있으므로 Phase 1은 너무 시끄럽지 않은 기본값으로 시작한다.

기본 필수 항목:

- 주문 내용
- 수량
- 희망일
- 수령 방식

조건부 필수 항목:

- 택배 선택 시 택배 주소

기본 필수에서 제외하지만 관리 모달에서 켤 수 있는 항목:

- 고객명
- 연락처
- 선물 용도
- 픽업 시간
- 알레르기
- 추가 옵션
- 고객 요청사항
- 사장님 내부 메모

## 확인과 정보 부족 표시

정보 부족과 확인필요는 같은 주의 레벨이지만 화면에서는 그룹을 분리한다.

반복 문장을 쓰지 않고 공통 안내 문구 아래 짧은 항목을 보여준다.

예:

```text
채워야 할 정보가 있어요
- 희망일
- 택배 주소

확인할 내용이 있어요
- 행사/답례품 주문
- 상품 후보 여러 개
- 수량 후보 여러 개
- 대량 기준 가능성
- 택배 가능 여부
```

목록에서는 대표 표시만 보여준다.

```text
채워야 할 정보 2개
확인할 내용 3개
```

상세에서는 짧은 항목 목록을 보여주고, 필요한 경우에만 원문 근거와 후보값을 펼친다.

## 화면 복잡도 제어

UI는 `DESIGN.md`의 프리미엄 운영 도구 톤을 따른다. 토스식 사용성 원칙을 참고해 한 화면에 한 가지 주 작업만 선명하게 둔다.

주문 수집 화면의 기본 흐름:

1. 원문 붙여넣기
2. 저장
3. 자동 추출 결과 확인
4. 부족하거나 애매한 항목만 짧게 수정

기본 목록은 다음 정보 중심으로 유지한다.

- 고객명 또는 이름 없음
- 주문 내용
- 수량
- D-Day
- 주의 표시

다음 내부 정보는 기본 노출하지 않는다.

- 상품 후보 전체
- 수량 후보 배열
- 메뉴 매칭 점수
- 실수량 계산 근거
- 파싱 규칙 이름

복잡한 정보는 사장님이 값을 수정하거나 확인 항목을 펼칠 때만 보여준다.

## 수동 수정값 우선

사장님이 직접 수정한 필드는 자동 추출값보다 우선한다.

원문 수정이나 재파싱으로 새 후보가 생겨도 수동 수정값을 덮어쓰지 않는다. 값이 다르면 필드 옆에 무채색 점을 표시하고, 터치/hover/focus 시 다음처럼 보여준다.

```text
원문에서는 "대추야자 2구/9구"로 보여요
```

이 표시는 경고가 아니라 참고 정보다.

## 중복 처리

중복 가능성은 저장 차단이 아니다.

Phase 1은 같은 원문 또는 공백/기호 차이만 있는 거의 같은 원문을 우선 감지한다. 중복 가능성이 있으면 확인할 내용에 `중복 가능성`을 추가한다. 실제로 중복 주문인지 보완 메시지인지는 사장님이 판단한다.

유사도 기반 중복 감지는 실제 원문 데이터가 쌓인 뒤 조정한다.

## 오류 처리

파싱 실패는 오류가 아니다.

- 원문이 비어 있지 않으면 저장한다.
- 추출하지 못한 필드는 비워둔다.
- 필수 필드가 비어 있으면 정보 부족으로 표시한다.
- 후보가 여러 개이거나 충돌하면 확인할 내용으로 표시한다.
- 저장을 막는 조건은 원문 없음 정도로 제한한다.

## 테스트 기준

도메인 테스트:

- 주문서형 라벨 추출
- 상담/의사표시형 상품 후보 추출
- 용도 키워드 추출
- 상품 구성 숫자와 주문 수량 분리
- 수량 후보 여러 개 처리
- 실수량 40구 이상 대량 판정
- 최소 주문 조건 판정
- 택배 가능 여부 확인 항목 생성
- 택배 주소 정보 부족
- 픽업 시간 미필수 처리
- 명시 날짜 D-Day 표시
- 상대 날짜 확인필요 처리
- 수동 수정값 보호
- 중복 가능성 경고

UI 테스트:

- 상품 후보와 수량 후보가 목록에 과도하게 노출되지 않는다.
- `채워야 할 정보가 있어요`와 `확인할 내용이 있어요`가 분리되어 보인다.
- D-Day 배지에서 원래 날짜를 확인할 수 있다.
- 수동 수정값과 재파싱 후보 차이가 무채색 참고 표시로 보인다.

## 후속 범위
