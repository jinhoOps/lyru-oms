# 주문 표준화 MVP 설계

## 목적

Lyru OMS의 첫 MVP는 제작/출고 관리 화면을 바로 만드는 것이 아니라, 여러 채널에서 들어오는 주문과 문의를 한곳에 모아 사장님이 빠르게 훑어볼 수 있게 만드는 것이다.

현재 카카오톡 채널과 인스타그램 DM은 고객이 주문서나 요청사항을 메시지로 남기는 형태다. 네이버 스마트스토어와 네이버예약은 더 구조화된 주문 소스일 가능성이 있지만, 실제 입력 형태는 추가 확인이 필요하다. 따라서 첫 MVP는 채널 자동 연동이나 완성된 주문 원장보다, 붙여넣은 주문/문의 텍스트를 가능한 만큼 구조화하고 빠진 정보를 드러내는 데 집중한다.

## 범위

### 포함

- 카카오톡, 인스타그램, 네이버 톡톡 등에서 받은 주문/문의 텍스트 붙여넣기
- 붙여넣은 원문 저장
- 알려진 주문서 라벨과 필드별 연관 키워드 기반 필드 추출
- 추출된 필드의 수동 수정
- 정보 부족과 확인 필요 사유 분리 표시
- 정보 부족과 확인 필요를 같은 주의 경고 레벨로 표시
- 수량 기반 대량 주문 확인 필요 표시
- 중복 원문 가능성 경고
- 수동 수정값 우선과 재파싱 차이 무채색 점 표시
- 정보 부족 판정에 사용할 필수 항목 관리 모달
- 사장님에게 확인할 운영 질문 임시 쪽지 화면
- 고객 요청사항과 사장님 내부 메모 분리
- 주문/문의 목록에서 출처, 희망일, 주문 요약, 확인 상태 확인
- 정보 부족 주문에서 원문 펼쳐보기

### 제외

- 구글폼 기반 고객 주문서
- 카카오톡/인스타그램 DM 자동 수집
- 네이버 스마트스토어 API 자동 연동
- 제작 준비, 출고 준비, 발송 완료 등 제작/출고 워크플로
- 정식 RDB 원장 설계
- 고객 확인 메시지 자동 생성
- 결제, 정산, 재고, 매출 분석

## 사용자 흐름

1. 사장님이 카카오톡, 인스타그램, 네이버 톡톡 등에서 받은 주문서나 문의 메시지를 복사한다.
2. Lyru OMS의 주문 수집 화면에서 출처를 선택하고 원문을 붙여넣는다.
3. 시스템은 알려진 라벨과 필드별 연관 키워드를 기준으로 가능한 필드를 자동 추출한다.
4. 추출된 값은 수정 가능한 입력칸에 채워진다.
5. 추출하지 못한 내용은 원문에 그대로 남는다.
6. 기본 필수 정보가 비어 있으면 `정보 부족` 사유를 남긴다.
7. 대량 주문 또는 맞춤 요청처럼 보이는 주문은 `확인필요` 사유를 남긴다.
8. `정보 부족` 또는 `확인필요` 사유가 하나라도 있으면 상태는 자동으로 `확인필요`가 된다.
9. `정보 부족`과 `확인필요`는 사유를 분리하되 같은 주의 경고 레벨로 보여준다.
10. 사장님은 정보 부족 주문에서 원문을 펼쳐 근거를 확인한 뒤 누락 정보를 고객에게 물어보거나 수동으로 보완한다.
11. 사장님은 자동 판정 사유가 없어도 필요하면 주문/문의를 직접 `확인필요`로 바꿀 수 있다.
12. 같거나 비슷한 원문이 이미 있으면 중복 가능성을 경고하되 저장은 허용한다.
13. 원문 수정이나 재파싱으로 새 값이 추출되어도 사장님이 수동 수정한 값은 자동으로 덮어쓰지 않는다.
14. 수동 수정값과 새 추출값이 다르면 무채색 점 표시로 차이를 알려준다.
15. 필요한 정보가 충분하다고 판단하면 상태를 `정리 완료`로 바꾼다.

## 데이터 모델

첫 MVP의 데이터는 완성된 주문 DB가 아니라 주문/문의 수집함이다. 제작과 출고 상태는 포함하지 않는다.

| Field | 설명 |
|-------|------|
| `source` | 카카오톡, 인스타그램, 네이버 톡톡, 네이버 스마트스토어, 네이버예약, 기타 |
| `rawText` | 붙여넣은 원문 |
| `customerName` | 성함 |
| `phone` | 연락처 |
| `orderItems` | 주문 내용 |
| `quantity` | 수량 |
| `purpose` | 선물 용도 |
| `fulfillmentType` | 픽업 또는 택배 |
| `desiredDateTime` | 픽업/택배 희망 날짜와 시간 |
| `address` | 택배 주소 |
| `allergyNote` | 견과류 알레르기 유무 |
| `options` | 보자기, 노리개, 꽃 등 추가 옵션 |
| `customerRequestNote` | 고객이 남긴 기타 요청사항 |
| `ownerMemo` | 사장님 내부 메모와 비고 |
| `missingFields` | 필수로 설정되어 있지만 비어 있거나 파싱이 안 된 항목 |
| `reviewReasons` | 정보 부족, 대량 주문, 맞춤 요청 등 확인 사유 |
| `warningLevel` | 정보 부족과 확인 필요를 같은 수준으로 묶어 보여주는 주의 경고 레벨 |
| `status` | 수집, 확인필요, 정리 완료 |
| `createdAt` | 생성 시각 |
| `updatedAt` | 수정 시각 |

### 메모 분리 원칙

`customerRequestNote`와 `ownerMemo`는 반드시 분리한다.

- `customerRequestNote`: 고객이 말한 요청사항이다. 예: "부모님 선물이라 고급스럽게 부탁드려요."
- `ownerMemo`: 사장님만 보는 내부 비고다. 예: "단골 고객", "가격 별도 안내 필요".

고객에게 다시 보여줄 수 있는 메시지나 출고 전 확인에는 `customerRequestNote`만 사용한다. `ownerMemo`는 고객-facing 문구에 포함하지 않는다.

상세/수정 화면에서는 고객 요청사항과 사장님 내부 메모를 완전히 별도 입력 영역으로 둔다. 목록에서는 고객 요청사항 여부와 내부 메모 존재 여부를 볼 수 있지만, 내부 메모 내용은 목록에 직접 노출하지 않는다.

## 상태

| 상태 | 의미 |
|------|------|
| `수집` | 원문을 붙여넣고 저장한 상태 |
| `확인필요` | 정보 부족이나 확인필요 사유가 하나라도 있어 사장님 확인이 필요한 상태 |
| `정리 완료` | 사장님이 필요한 정보가 충분하다고 표시한 상태 |

제작/출고 상태인 `제작 준비`, `출고 준비`, `발송 완료`는 이번 MVP에 넣지 않는다. 주문 수집과 제작 관리를 섞지 않고, 실제 사장님 검증 후 다음 단계에서 방향을 정한다.

`정리 완료`는 사장님이 부족하거나 애매한 정보를 수동으로 확인했다는 의미다. 따라서 필수 항목 설정, 대량 주문 기준, 주문 필드가 나중에 바뀌어도 기존 `정리 완료` 주문/문의는 자동으로 `확인필요`로 되돌리지 않는다.

## 파싱 규칙

MVP 파싱은 완벽한 자동화를 목표로 하지 않는다. 원문을 잃지 않고 사람이 빨리 정리하도록 돕는 것이 목표다.

Phase 1은 라벨 기반 추출로 시작한다. 자유 문장을 추정해서 값을 채우지 않고, 미리 정한 라벨 또는 연관 키워드가 있을 때만 해당 필드를 채운다.

연관 키워드 세트는 Phase 1에서 사장님이 직접 관리하지 않는다. 사장님에게 자주 쓰는 표현을 확인한 뒤 개발자가 코드나 문서에서 키워드 세트를 업데이트한다. 관리 모달은 필수 항목과 대량 주문 기준처럼 사장님이 운영상 바로 판단할 수 있는 설정에 집중한다.

우선 다음 라벨과 연관 키워드를 인식한다.

- `성함`
- `이름`
- `고객명`
- `연락처`
- `전화번호`
- `휴대폰`
- `주문 내용`
- `주문상품`
- `상품`
- `수량`
- `개수`
- `주문 수량`
- `선물 용도`
- `용도`
- `픽업/택배`
- `수령 방식`
- `수령방법`
- `픽업 날짜 및 시간`
- `픽업 시간`
- `방문 시간`
- `택배 주소`
- `주소`
- `배송지`
- `견과류 알레르기 유무`
- `알레르기`
- `추가 옵션`
- `보자기/노리개/꽃`

라벨과 연관 키워드를 찾지 못하거나 값이 애매하면 원문을 보존하고 해당 필드를 비워 둔다. 비어 있는 필수 항목은 `missingFields`에 기록한다.

원문 수정이나 재파싱 시 새 추출값이 나오더라도 사장님이 이미 수동으로 고친 값은 자동으로 덮어쓰지 않는다. 수동 수정값과 새 추출값이 다르면 강한 경고가 아니라 무채색 점 하나로 차이가 있음을 보여준다. 마우스를 올리거나 모바일에서 터치하면 "원문에서는 이렇게 가져왔어요" 문구와 새 추출값을 보여준다.

## 확인필요 판정

다음 조건 중 하나라도 해당하면 `확인필요`로 표시한다.

- 필수로 설정된 항목이 비어 있다.
- 수량이 대량 주문 기준 이상이다.
- 맞춤 요청으로 보이는 문구가 있다.
- 고객 요청사항이 있지만 어떤 필드에 넣어야 할지 애매하다.

기본 필수 항목은 고객명, 연락처, 주문 내용, 수량, 희망일, 수령 방식이다. 주소는 수령 방식이 택배일 때만 필수이고, 픽업 시간은 수령 방식이 픽업일 때만 필수다.

대량 주문 기준은 Phase 1에서는 단순 수량으로 판단한다. 상품 DB가 없고 9구 상품과 4~5구 상품처럼 상품별 생산 부담이 다르므로 완전한 기준은 아니지만, 초기 검증을 위해 기본값은 수량 5개 이상으로 둔다.

필수 항목과 대량 주문 기준 수량은 사장님이 관리 모달에서 나중에 조정할 수 있어야 한다. 맞춤 요청 키워드는 실제 주문 사례를 보며 조정한다. 초기값은 구현 단계에서 보수적으로 정한다.

`정보 부족`과 `확인필요`는 같은 경고 레벨로 묶어 보여주되, 상세 사유는 분리해서 남긴다. 예를 들어 한 주문이 연락처 누락과 맞춤 요청을 동시에 가지면 `정보 부족: 연락처`, `확인필요: 맞춤 요청`을 모두 표시한다.

정보 부족이나 확인필요 사유가 하나라도 있으면 주문/문의 상태는 자동으로 `확인필요`가 된다. 상태를 `수집`으로 둔 채 경고만 붙이지 않는다.

사장님은 자동 판정 사유가 없어도 필요하면 수동으로 `확인필요` 상태를 줄 수 있다. 이 조작은 주 흐름이 아니라 예외적 보조 기능으로 본다.

## 화면 구조

### 주문 수집 화면

- 출처 선택
- 원문 붙여넣기 입력
- 파싱 실행 또는 저장
- 추출 결과 미리보기
- 정보 부족 표시

### 필수 항목 관리 모달

- 정보 부족 판정에 사용할 항목 선택
- 대량 주문으로 볼 기준 수량 설정
- 택배 주소처럼 특정 수령 방식에서만 필수인 조건부 항목 확인
- 픽업 시간처럼 특정 수령 방식에서만 필수인 조건부 항목 확인
- 설정 변경 후 새로 수집되는 주문/문의에 적용
- 기존 `정리 완료` 주문/문의는 설정이나 필드 수정 때문에 자동으로 `확인필요`로 되돌리지 않음

### 주문/문의 목록

목록은 한눈에 훑는 용도다. 원문 전체는 기본 목록에 노출하지 않고, 1차 필드는 다음으로 둔다.

- 출처
- 고객명
- 희망일
- 수령 방식
- 주문 요약
- 고객 요청사항 여부
- 내부 메모 존재 여부
- 정보 부족 여부
- 확인 필요 여부
- 상태

정보 부족과 확인 필요는 같은 주의 경고 레벨로 묶어서 우선 확인 대상으로 보이게 한다. 다만 뱃지나 상세 사유에서는 두 사유를 구분한다.

### 사장님 확인 질문 임시 쪽지 화면

개발 방향을 정하기 위해 사장님에게 확인해야 할 질문을 주문 수집 화면 상단의 접이식 임시 쪽지 영역에 모아둔다. 사장님은 개발 지식이 없는 일반 사용자로 가정하고, 화면 문구는 기술 용어가 아니라 실제 운영 언어로 쓴다. 이 화면은 정식 운영 기능이 아니라 관리 모달 기본값과 화면 문구를 정하기 위한 검증 도구다.

답변 입력이나 저장 기능은 만들지 않는다. 사장님은 쪽지의 질문을 보고 개발자에게 직접 연락해 답변을 전달한다.

쪽지는 주문 수집 작업을 막지 않아야 하므로 모달로 띄우지 않는다. 기본 상태와 접힘 상태는 구현 시 화면 밀도를 보고 정하되, 사장님이 닫거나 접을 수 있어야 한다.

초기 질문 예시는 다음과 같다.

- 몇 개부터 미리 확인해야 하는 큰 주문으로 보시나요?
- 택배 주문에서 꼭 빠지면 안 되는 정보는 무엇인가요?
- 픽업 주문에서 꼭 빠지면 안 되는 정보는 무엇인가요?
- 주문 메시지에서 자주 빠지는 정보는 무엇인가요?
- 고객님들이 이름, 연락처, 주소 같은 정보를 보통 어떤 표현으로 적어주시나요?
- 어떤 표현이 있으면 맞춤 요청으로 따로 확인해야 하나요?

정보 부족 주문은 원문을 펼쳐볼 수 있어야 한다. 다만 모든 주문의 원문 전체를 목록에 상시 노출하지 않는다.

목록 필드는 실제 사장님과 화면을 보며 추가 조정한다.

### 상세/수정 화면

- 원문
- 추출된 주문 필드
- 고객 요청사항
- 사장님 메모/비고
- 누락 필드
- 확인 필요 사유
- 상태 변경

## 오류 처리

- 파싱 실패는 저장 실패가 아니다.
- 알 수 없는 양식도 `rawText`만으로 저장할 수 있어야 한다.
- 필수 정보가 빠져도 저장은 가능하지만, 목록과 상세에서 분명히 표시한다.
- 필수 정보가 빠지거나 파싱이 실패해도 원문 저장은 차단하지 않는다.
- 같거나 비슷한 원문이 있어도 저장은 가능하지만, 중복 가능성을 분명히 표시한다.
- 파싱된 값은 언제든 수동 수정할 수 있어야 한다.
- 수동 수정값은 재파싱 결과보다 우선하며 자동으로 덮어쓰지 않는다.
- 재파싱 결과와 수동 수정값이 다르면 무채색 점으로 표시하고, 터치/호버 시 "원문에서는 이렇게 가져왔어요"와 새 추출값을 보여준다.
- `정리 완료` 주문/문의의 필드를 수정해도 상태는 자동으로 바뀌지 않는다.
- 고객 요청사항과 사장님 메모는 저장, 표시, 향후 메시지 생성에서 섞지 않는다.

## 검증 기준

- 공유된 주문서 양식을 붙여넣으면 주요 필드가 자동으로 채워진다.
- 필수 항목이 빠진 주문서도 원문이 보존되고 `정보 부족`으로 표시된다.
- 필수 항목이 빠지거나 파싱이 실패해도 원문은 저장된다.
- `정보 부족` 주문에서는 원문을 펼쳐볼 수 있지만, 원문 전체가 목록에 항상 노출되지는 않는다.
- 고객 요청사항과 사장님 메모를 별도로 입력하고 수정할 수 있다.
- 목록에서 출처, 희망일, 주문 요약, 정보 부족, 확인 필요 여부를 빠르게 확인할 수 있고 둘은 같은 주의 경고 레벨로 보인다.
- 모바일 폭에서도 붙여넣기, 목록 확인, 상세 수정 흐름이 무리 없이 동작한다.
- 제작/출고 관리 기능 없이도 사장님과 함께 주문 수집 화면의 유용성을 검증할 수 있다.

## 다음 검증 질문

첫 화면을 실제 사장님과 함께 볼 때 다음을 확인한다.

- 목록에서 더 필요한 필드가 있는가?
- 관리 모달에서 필수 항목과 대량 주문 기준 수량을 조정할 때 사장님이 이해하기 쉬운 항목명은 무엇인가?
- 대량 주문 기준 기본값 5개 이상이 실제 운영에 맞는가?
- 맞춤 요청으로 분류해야 하는 표현은 무엇인가?
- 네이버 스마트스토어와 네이버예약 데이터는 어떤 형태로 받을 수 있는가?
